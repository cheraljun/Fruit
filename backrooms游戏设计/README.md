## 设计核心原则

### 1. 叙事优先
- 用文字描写营造氛围，不依赖数值
- 理智不是"73/100"，而是"你感到墙纸在蠕动"
- 杏仁水不是"消耗品ID:001"，而是"希望的象征"

### 2. 选择驱动
- 每个选择都有trade-off
- 没有"最优解"
- 遭遇实体是"做决定"，不是"战斗"

### 3. 结合编辑器
- 热区系统 → 枢纽探索
- jumpToNode → 空间导航
- 世界地图 → 快速返回
- Blockly → 状态门槛和条件判断

### 4. 忠于世界观
- 基于`backrooms世界体系/`
- 层级、实体、物资系统保持一致
- 氛围恐怖，不是Jumpscare

## 关键设计要点

### Level 4是核心
- 玩家的"家"
- 无限杏仁水补给
- 热区枢纽设计
- 所有路线的中转站

### 资源压力不是数值
- 用状态标记（sufficient/low/critical）
- 用文字描写（"杏仁水不多了"）
- 不显示UI数值条

### 结局是变量累积
- 不是单一选择
- 整个游戏的多个决策汇聚
- karma_score等变量驱动

### 实体遭遇是模板
- Smiler：3选项教学遭遇
- Hound：连续追逐序列
- Level Fun：充分警告的陷阱

## 节点数量估算

```
Level 0: 15个节点（教学）
Level 1: 20个节点（探索+遭遇）
Level 2: 10个节点（过渡）
Level 4: 15个节点（枢纽+隐藏）
Level 5: 15个节点（The Beast）
Level 11: 20个节点（定居）
Level Fun: 10个节点（陷阱）
实体遭遇: 30个节点（各种实体）
结局: 15个节点（8种结局）
过渡/系统: 10个节点

总计: 约160个节点
```

## 开发建议

### 按模块开发
1. **先做Level 0+1** - 建立基础循环
2. **再做Level 4** - 枢纽系统
3. **然后做实体遭遇** - 紧张感
4. **最后做结局** - 收尾

### 原型验证
- 先做20个节点的核心循环
- 测试玩法是否有趣
- 再扩展到完整内容

### 内容复用
- 创建模板节点
- 复制修改
- 保持风格一致

## 最终检查清单

在完成设计后，问自己：

✅ 这能用mo编辑器实现吗？  
✅ 这符合Backrooms氛围吗？  
✅ 玩家的选择真的有意义吗？  
✅ 这是叙事游戏还是数值游戏？  

如果所有答案都是"是"/"叙事游戏"，那就对了。
